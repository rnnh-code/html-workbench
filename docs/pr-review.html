PR #1834 6 files changed focus: services/limiter

Move rate limiter from in-process counters to Redis-backed counters

Replaces local mutex / counter state with Redis SETNX-based locking and (per the file map) Redis-backed counters. Review focus per author: services/limiter/redis_lock.go — the lock acquisition logic.

Recommended action
Request changes
The visible AcquireLock change has at least one correctness blocker (no lock owner, so any caller — or any retry — can release someone else's lock) and one safety blocker (no ReleaseLock shown alongside; if it uses plain DEL, the limiter is unsafe under contention). Fix the lock primitive before merging.

Top concerns severity-tagged · click a chip in the sidebar to filter

  • Blocker Lock has no owner token — any caller can release any lock redis_lock.go · L18+

    SetNX(ctx, "lock:"+key, "1", ttl) stores the literal value "1" for every acquirer. A correct Redis lock requires a unique per-acquirer token (UUID/nanoid) stored as the value, and the release path must only DEL when the value still matches its token — typically via a Lua CAS script (if GET == token then DEL). Without an owner token, a slow process whose TTL has expired and been re-acquired by another process can later DEL the new owner's lock. This is the canonical Redis-lock footgun. The release path is not in the abridged diff, so this needs verification, but the value being a constant "1" is enough to flag it as a blocker on the acquire side.

  • Blocker Bare error return turns transient Redis failures into "lock unavailable" redis_lock.go · L20–22

    if err != nil { return false, err } conflates two very different outcomes: "someone else holds the lock" and "Redis is unreachable / context cancelled / timeout". Callers in services/limiter/limiter.go (not shown) need to know which one they got — for fail-open vs fail-closed behavior on the rate limiter. Recommend wrapping the error and either (a) returning a typed error the caller can switch on, or (b) treating ctx.Err() and connection errors as a distinct "limiter unavailable" mode with explicit policy. Today's behavior risks silently dropping all rate-limiter requests during a Redis blip.

  • Major Signature didn't change — but semantics did. Boolean return now means something different. redis_lock.go · L18

    localMutex.TryLock returned a boolean reflecting only mutex contention. redis.SetNX ok-flag means "key did not previously exist," which depends on TTL behavior, clock skew, and Redis availability. Same signature, different invariants. All callers in limiter.go need a fresh look. Worth either renaming (AcquireDistributedLock) or documenting the new failure modes in the doc comment so the next reader is not misled.

  • Major No retry / backoff strategy visible — thundering-herd risk on contended keys redis_lock.go · whole function

    With an in-process mutex, contention was cheap; with Redis SETNX, every losing acquirer hits the network and burns a request. If callers in limiter.go spin-retry without jittered backoff, a hot key (popular endpoint, abusive client) can saturate Redis. Need to confirm caller behavior and ideally add a small randomized backoff at this layer. Caller code not in the abridged diff — needs verification.

  • Major Tests must cover the contended path, not just happy-path acquire lock_test.go

    Test file contents not in the abridged diff — needs verification. Required cases for a distributed lock: (1) two goroutines, only one acquires; (2) TTL expiry releases the lock; (3) Redis-down returns the typed error and does not falsely report success or failure; (4) release-after-expiry does NOT delete a re-acquired lock (the canonical bug). If any of these are missing, this PR should not merge.

  • Minor Hard-coded "lock:" prefix — promote to a constant redis_lock.go · L19

    String-concatenated key prefixes drift across files over time. Move "lock:" to a package-level const lockKeyPrefix (and ideally a lockKey(key) helper) so the limiter, debug tools, and any ops scripts agree on the namespace. Also opens the door to env-prefixed keys for staging/prod isolation in a shared Redis.

  • Minor Validate ttl > 0 at the boundary redis_lock.go · L18

    A zero or negative ttl on SetNX behaves differently from the old in-process mutex (which had no concept of TTL). Reject it explicitly with a typed error to avoid permanently-held locks from a misconfigured caller.

  • Nit Single-statement if err != nil on one line is fine, but inconsistent with the rest of the package redis_lock.go · L21

    Style only — match the surrounding files. Worth a glance, not a blocker.

  • Praise Keeping the function signature stable is the right call redis_lock.go · L18

    Not bumping the signature shrinks the diff at every call site and isolates the risk to the body. Good instinct. Pair it with the doc-comment update from concern #3 and this becomes a clean refactor.
