Implementation plan

Auth migration: session cookies to JWT

Phased plan to move web, mobile, and three internal services off session cookies onto a JWT-based scheme over six weeks, without breaking existing sign-ins.

Timebox: 6 weeks Surfaces: web, mobile, orders, billing, notifications Owner: TBD Status: draft

TL;DR

What we're doing, on what cadence, and why it's safe to run live.

Approach
Dual-mode: services accept both session cookies and JWTs during transition
Token shape
Short-lived access JWT + opaque rotating refresh token
Cutover style
Gradual flag-driven rollout per surface, not a big-bang switch
Rollback
Feature flag flips clients back to cookie auth; sessions kept warm until week 6
Hard prereq
Auth service issues + verifies JWTs before any client switches
Done when
Cookie code paths removed and traffic on JWTs is >99% for 7 days

Context

Why this work, and what it changes for users.

Today, every surface authenticates via server-managed session cookies issued by services/auth/. Mobile already pays a tax simulating cookie storage; the three internal services share session validation logic that's drifted slightly between them. Moving to JWT lets each service verify locally with a public key, removes the synchronous cookie lookup on every internal call, and gives mobile a first-class token model.

The migration is risky for one reason: auth touches every request. The plan is therefore additive first, subtractive last — we issue and accept JWTs alongside cookies, ramp traffic, and only remove cookie code after the new path is boring.

Phased rollout

Click a phase to expand. Each phase lists what changes, where in the codebase, and how we know it landed.

1
Foundation in the auth service (week 1–2)
Issue JWTs and publish JWKS without changing any client behaviour.
Owner: TBD
Risk: medium

Goals

  • Add a JWT signing key (asymmetric, e.g. RS256 or EdDSA) and a JWKS endpoint at services/auth/jwks.json.
  • Issue a short-lived access token (~5 min) and a rotating opaque refresh token on successful login.
  • Continue setting the existing session cookie unchanged; new tokens are returned in the login response body in addition.
  • Add a token introspection / verify helper used by internal services.

Code touchpoints

  • services/auth/: signing keys, key rotation hook, JWKS handler, refresh-token store, claims schema.
  • Shared lib (new): services/auth/lib/jwt-verify consumed by other services in phase 2.

Verification

  • Unit tests on issuer + verifier covering expiry, clock skew, wrong key, malformed token.
  • Staging smoke: log in, decode token, verify against JWKS, refresh once, revoke refresh.
  • No production traffic shift in this phase.
2
Backends accept JWT alongside cookies (week 2–3)
Orders, billing, and notifications start accepting either credential.
Owner: TBD
Risk: medium

Goals

  • Each service's request middleware looks for Authorization: Bearer first, falls back to session cookie.
  • Verify JWT signature against cached JWKS (cache TTL = 10 min, with key-rotation tolerance).
  • Map JWT claims to the same internal user identity object the rest of the code expects, so handlers don't change.
  • Emit a per-request metric labeled auth_method = jwt | cookie.

Code touchpoints

  • services/orders/ middleware + tests.
  • services/billing/ middleware + tests.
  • services/notifications/ middleware + tests.
  • Shared services/auth/lib/jwt-verify consumed by all three.

Verification

  • Each service's contract tests pass with both auth methods.
  • Synthetic probe sends one JWT request and one cookie request per minute against staging; both 200.
  • auth_method metric visible in dashboard with both labels in production traffic before phase 3.
3
Web client switches to JWT, ramped (week 3–4)
Web obtains and presents JWTs behind a feature flag, ramped 1% -> 100%.
Owner: TBD
Risk: high

Goals

  • Web stores the access token in memory and the refresh token in an HttpOnly; Secure; SameSite=Lax cookie scoped to the auth domain.
  • Authenticated fetches send Authorization: Bearer; refresh on 401 with a single in-flight refresh promise.
  • Feature flag auth.use_jwt ramps 1% → 10% → 50% → 100% with a 24h soak between steps.
  • SSR path also receives the JWT for downstream service calls.

Code touchpoints

  • apps/web/: auth client, fetch wrapper, SSR token forwarding, login + logout flows.
  • Flag config in the deployment system; named auth.use_jwt.

Verification

  • 401 rate stable or lower at each ramp step (compare 24h vs control cohort).
  • p95 latency on protected routes within +5ms of pre-ramp.
  • No spike in support tickets tagged login or signed-out.
  • Refresh-token rotation success rate >99.5%.
4
Mobile switches to JWT, staged release (week 3–5)
Mobile is slower because store review + slow updaters; plan for a long tail.
Owner: TBD
Risk: high

Goals

  • Mobile stores access token in memory and refresh token in OS keychain.
  • Network layer adds the bearer header; refresh on 401 with a single coalesced refresh.
  • Ship behind a remote config flag so we can fall back without a new build.
  • Staged store rollout: 5% → 20% → 50% → 100%, gated on crash + 401 rate.

Code touchpoints

  • apps/mobile/: auth module, network interceptor, secure storage, login + logout flows.
  • Remote-config flag in same naming scheme as web (auth.use_jwt).

Verification

  • Crash-free sessions match prior week within 0.1%.
  • 401 rate flat at each ramp step.
  • Refresh-token rotation success >99% (slightly lower bar than web due to flaky networks).
  • Telemetry shows old-version users still on cookie path are unaffected.
5
Deprecation gate (week 5)
Cookie path stays available, but is now considered legacy.
Owner: TBD
Risk: low

Goals

  • Auth service starts logging cookie-based authentications with a deprecated=true tag.
  • Internal services emit a low-rate warning log when they fall back to cookie verification.
  • Confirm auth_method = jwt is >99% across all three services for 7 consecutive days.

Code touchpoints

  • services/auth/: logging + dashboard panel.
  • services/orders/, services/billing/, services/notifications/: warning log on fallback.

Verification

  • Dashboard shows JWT share >99% across each service for 7 days.
  • No support escalation tied to the deprecation log noise.
  • Stakeholders sign off on removal.
6
Cookie removal and cleanup (week 6)
Subtractive step. Only run after the phase 5 gate passes.
Owner: TBD
Risk: medium

Goals

  • Remove cookie issuance from services/auth/.
  • Remove cookie verification fallback from each backend service.
  • Drop the auth.use_jwt flag.
  • Decommission the session store after a 7-day grace period of read-only mode.
  • Update onboarding docs and the runbook.

Code touchpoints

  • services/auth/ session module deleted.
  • Cookie middleware deleted from each backend service.
  • apps/web/ + apps/mobile/ dead branches removed.

Verification

  • Repo grep for old session helpers returns zero hits.
  • End-to-end auth tests pass on a build with the cookie path stripped.
  • Production stable for 48h after removal before flag config is deleted.

Risk register

Filter by severity to see what's actually scary versus what's just paperwork.

Severity Risk Why it matters Mitigation
Critical Mass sign-out at cutover Mismatched flag state across services could invalidate live sessions. Dual-accept window; ramp via flag, not deploy; keep session store warm until phase 6 verification passes.
Critical Signing-key compromise Anyone with the private key can mint user identities. Asymmetric keys; key stored in KMS; rotation tested in phase 1; revoke + rotate runbook ready before phase 3.
Warning Refresh-token replay Stolen refresh token could mint new access tokens indefinitely. Rotating refresh tokens with reuse detection; on detected reuse, revoke entire token family and force re-login.
Warning Mobile long tail on old builds Some users won't update for weeks; cutting cookies early signs them out. Don't remove cookie path until JWT share >99% for 7 days. Communicate force-update if tail stalls.
Warning Clock skew on backend hosts JWT exp / nbf rejections cause user-visible 401s. Allow 60s skew tolerance in verifier; ensure NTP is healthy on all hosts; alarm on skew.
Warning Token bloat on requests If we stuff roles/permissions into JWT, header size balloons and breaks proxies. Keep JWT claims minimal (sub, exp, iat, sid, aud); fetch authorization details server-side.
Info JWKS cache thundering herd If cache expires together across services on key rotation, JWKS endpoint spikes. Stagger cache TTL; fetch with jitter; pre-warm on rotation event.
Info Logging tokens in error traces Authorization headers can leak into stack traces and logs. Redact Authorization header in logger middleware; add a unit test for redaction.
Info Logout revocation semantics Pure stateless JWT means logout doesn't actually invalidate the access token until it expires. Short access-token lifetime (5 min); refresh revocation list; document the trade-off in the runbook.

Out of scope

Things this plan deliberately does not do.

Identity provider swap. We are not moving to a third-party IdP (Auth0, Cognito, etc.) in this window.
OAuth2 / OIDC for third parties. External integrations stay on their existing model.
SSO / SAML. Untouched.
Permissions model rework. Roles and scopes stay the same; only the credential format changes.
MFA changes. The MFA flow stays where it is; it now hands off to JWT issuance instead of session creation.
Public API tokens. Service-to-service and partner API keys are a separate workstream.
