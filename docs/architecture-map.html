Request lifecycle — "Place order"

What happens when a customer hits "Place order" on the storefront or mobile app. Steps 1–5 are synchronous on the request path; steps 6–8 are async via the queue.

  1. Client submits order

    Web or mobile POSTs cart, address, and a Stripe payment-method id to orders.

    apps/web · apps/mobile

  2. Reserve inventory

    orders calls inventory to reserve stock for each line item. Failure here aborts the order before any money moves.

    orders → inventory

  3. Charge payment

    orders calls payments, which creates a PaymentIntent on Stripe and confirms it.

    orders → payments → Stripe

  4. Persist order

    orders writes the order row + line items to its Postgres schema, transitioning state to confirmed.

    orders → PG (orders)

  5. Respond to client

    orders returns the order id and confirmation summary. Client renders a receipt page.

    orders → client

  6. Emit order.placed

    orders enqueues an order.placed event onto Redis.

    orders → Redis

  7. Commit reservation

    inventory consumes the event and converts the reservation into a permanent stock decrement.

    inventory ← Redis

  8. Send notifications

    notifications consumes the event and dispatches a confirmation email via SendGrid and an SMS via Twilio.

    notifications → SendGrid · Twilio

Glossary & assumptions

What the labels mean (click to expand)
Per-service Postgres schema
Each backend service has its own logical schema in Postgres. Services do not read each other's tables — cross-service data goes over HTTP or via the Redis event queue.
Reserve vs. commit (inventory)
"Reserve" is a soft hold placed during checkout to prevent overselling. "Commit" is the permanent stock decrement, applied after the order is confirmed.
order.placed event
An event written to a Redis-backed queue after the order is confirmed in Postgres. Currently consumed by inventory (to commit the reservation) and notifications (to send confirmations).
Stripe wrapping
services/payments is the only service that talks to Stripe. Other services request charges or refunds through it — they never hold Stripe credentials.
Assumption: shared-types is build-time only
packages/shared-types exports TypeScript types only. Both client apps depend on it; the Go and Node services do not consume it directly. Treat the "Shared" team as a shared platform responsibility unless told otherwise.
